# 根因修复复盘

## 边界

- 默认只读取证据并输出复盘结论，不修改代码；用户明确要求继续修复时再实施修改。
- 以失败测试、日志、原始症状、当前差异和相关实现为证据。证据不足时明确标记未知项，不补全未经证实的失败故事。

## 工作流

1. 收集复盘输入。
   确认原始症状、可复现证据、当前实现差异、相关测试和验证结果。缺少关键输入时先界定能够确认的范围。

2. 从第一性原理重建失败链路。
   识别触发条件、可观察症状、最早被破坏的不变量，以及本应维护该不变量的模块。

3. 区分根因和症状。
   将日志、错误码、UI 状态、重试行为视为证据，而不是解释本身。不要停在错误显现的位置，要继续追溯到更早的所有权、生命周期、契约或状态模型问题。

4. 明确系统应维护的不变量。
   用一句话写出修复后必须一直成立的规则。例如：“每个播放会话独占一个新的渲染 Surface”、“认证状态只有一个事实来源”、“解析要么显式成功，要么显式失败，不产生部分默认值”。

5. 检查当前实现是否维护该不变量。
   查找临时特判、重复状态、并行验证、隐藏回退、宽泛异常捕获、沉默默认值、重试或延迟补偿，以及上层模块泄漏底层实现细节的情况。

6. 将修复收敛到不变量的所有者。
   优先建议明确生命周期、资源边界、类型化状态和统一契约，并删除被新不变量取代的 guard、fallback、重复状态和过时分支。

7. 验证问题类别，而不只是验证单个复现路径。
   除原始复现外，至少验证一个相邻场景，证明它因为新不变量而安全，而不是因为某个具体特判刚好命中。

8. 形成可追溯结论。
   引用具体文件和行号，并将结论区分为已确认根因、结构性风险和待验证假设。

## 输出要求

复盘结果应包含：

- 根因：最早导致该 Bug 存在的结构性原因。
- 当前实现问题：补丁式、位置错误或边界不清的逻辑。
- 新不变量：防止同类问题复发的规则。
- 修改或建议：聚焦删除、所有权收敛和边界修正。
- 验证方式：证明该不变量成立的命令、测试或手动场景。
- 证据状态：区分已确认事实、风险和待验证假设。

如果当前实现已经足够结构化，也要明确说明它依赖的不变量是什么。
