# 根因修复复盘

## 工作流

1. 从第一性原理重建失败链路。
   识别触发条件、可观察症状、被破坏的不变量，以及本应维护该不变量的模块。

2. 区分根因和症状。
   将日志、错误码、UI 状态、重试行为视为证据，而不是解释本身。不要停在错误显现的位置，要继续追溯到更早的所有权、生命周期、契约或状态模型问题。

3. 检查当前修复是否是补丁式实现。
   重点查找：字符串驱动逻辑、按 codec/type/user-agent/path 的临时特判、重复状态、并行验证、隐藏回退、宽泛 try/catch、沉默默认值、重试循环、延迟等待，以及 UI 层了解底层实现细节。

4. 明确系统应维护的不变量。
   用一句话写出修复后必须一直成立的规则。例如：“每个播放会话独占一个新的渲染 Surface”、“认证状态只有一个事实来源”、“解析要么显式成功，要么显式失败，不产生部分默认值”。

5. 将修复移动到不变量的拥有者。
   优先使用生命周期所有权、明确资源边界、类型化状态、统一契约和删除过时分支。若行为依赖底层实现细节，应暴露类型化契约或把决策下沉到真正拥有该细节的模块。

6. 删除临时兜底层。
   删除被新不变量取代的 guard、fallback 或特判。保留资源所有者应做的清理逻辑，但移除只用于掩盖非法状态的补偿动作。

7. 验证问题类别，而不只是验证单个复现路径。
   除原始复现外，至少验证一个相邻场景，证明它因为新不变量而安全，而不是因为某个具体特判刚好命中。

## 输出要求

复盘结果应包含：

- 根因：最早导致该 Bug 存在的结构性原因。
- 当前实现问题：补丁式、位置错误或边界不清的逻辑。
- 新不变量：防止同类问题复发的规则。
- 修改或建议：聚焦删除、所有权收敛和边界修正。
- 验证方式：证明该不变量成立的命令、测试或手动场景。

如果当前实现已经足够结构化，也要明确说明它依赖的不变量是什么。
