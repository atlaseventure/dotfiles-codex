# 结构化重构代码库

以行为保持为前提治理结构问题。先用证据确认职责、重复和依赖边界，再实施可验证的拆分；不要按行数机械切文件，也不要为展示 SOLID 强行增加接口或层次。

## 工作流

### 1. 确认约束与现场

1. 读取仓库及子目录适用的指令、架构文档、生成规则和验证入口。
2. 检查工作区、暂存区和近期历史，区分当前任务修改、用户已有修改及生成产物。
3. 明确对外契约：API、CLI、数据库、序列化格式、公开类型、事件、日志和文件路径。
4. 未经用户要求，不提交、不推送、不顺手修改无关行为。

### 2. 建立基线

1. 运行当前最相关的测试、lint、类型检查或构建命令。
2. 记录已有失败；不要把基线失败误归因于重构，也不要通过放宽检查隐藏它。
3. 对生成文件先运行或定位生成器，禁止直接把生成内容当成权威源修改。

### 3. 扫描结构信号

运行只读扫描脚本：

```bash
bash <skill-dir>/scripts/scan-codebase.sh <repo-root> 40
```

结合代码搜索、调用关系和测试判断候选项。至少检查：

- 文件规模、函数规模、类型数量和依赖数量；
- 一个文件是否包含查询解析、业务编排、持久化、映射、传输、审计等多种变化原因；
- 相同验证、转换、响应、错误处理或协议逻辑是否由多个消费者重复维护；
- 单目录是否混合多个领域、层次或命名规则；
- `utils`、`common`、`model`、`service`、`index` 等含糊名称是否掩盖真实职责；
- 测试、迁移、静态数据和生成文件是否只是天然较长，而非职责失控。

不要仅凭行数、文件数量或目录深度判定问题。

### 4. 排定重构目标

优先选择同时满足多项条件的目标：

- 有两个以上独立变化原因；
- 重复逻辑已有多个真实消费者，或即将持续扩散；
- 依赖方向混乱、命名误导或修改经常跨越无关区域；
- 存在覆盖当前行为的测试，或能够先补充低成本回归测试；
- 拆分后能形成稳定、可命名的所有者，而非产生更多转发层。

明确列出不拆分项及理由，例如“生成文件”“单一映射表”“共享一次性测试环境”“虽然较长但职责内聚”。

### 5. 设计职责与依赖边界

将 SOLID 转成可检查的结构决策：

- **SRP**：按变化原因拆分，例如入口、请求解析、业务编排、上传、下载、响应映射、查询、持久化。
- **OCP**：只有稳定扩展点确实存在时才引入策略或注册机制；不要用抽象替代简单分支。
- **LSP**：接口实现必须保持错误、状态和副作用语义；不能用沉默 fallback 伪造可替换性。
- **ISP**：在消费者一侧定义窄接口；避免让调用方依赖庞大仓储或服务接口。
- **DIP**：低层共享组件不得反向依赖具体业务或传输层；领域层不得导入 HTTP/UI 实现。

抽取复用组件时同时满足：

1. 至少两个真实调用方，或一个明确且迫近的第二调用方；
2. 规则稳定且应由单一所有者维护；
3. API 比复制逻辑更容易理解；
4. 名称表达具体能力，例如 `filetransfer`、`requestquery`，避免泛化为 `helpers`；
5. 抽取后依赖方向更清晰，不产生循环依赖。

### 6. 规划目录与命名

1. 优先沿既有架构分层；新建子目录必须代表稳定边界，而不是只为减少文件数量。
2. 按领域拆分业务代码，按协议或技术能力放置跨领域基础组件。
3. 保持目录浅而明确；通常一到两层即可，不创建只有一个转发文件的层。
4. 文件名使用“领域 + 角色”或明确角色，例如 `device_log_download.go`、`model_repair.go`、`query_parser.ts`。
5. 同步重命名对应测试、引用、生成配置、trace source 和文档路径。

### 7. 小步实施

1. 先移动纯类型、纯函数或查询解析，再移动包含副作用的流程。
2. 每次拆分保持编译通过，尽量保持公开符号、错误值、响应状态、响应体和副作用顺序不变。
3. 复用组件先补边界测试，再替换调用方。
4. 删除确认无引用的旧函数、重复常量和失效文件；不要保留两份可独立漂移的实现。
5. 避免混入全仓格式化、行为优化或契约变更；发现真实缺陷时单独说明并评估是否应另行修复。

### 8. 验证行为保持

按风险选择并运行：

1. 新组件和被拆模块的定向测试；
2. 全量单元测试、lint、类型检查和构建；
3. race、集成或端到端测试；
4. schema、客户端或文档重新生成，并确认生成差异符合预期；
5. `git diff --check`、工作区状态和完整差异复核。

若验证失败，追踪最早被破坏的不变量并修复根因。不要吞错、跳过测试或增加沉默默认值来让重构“通过”。

### 9. 交付结果

报告以下内容：

- 拆分前后的职责与文件规模变化；
- 抽取组件、真实消费者和依赖方向；
- 目录及文件重命名结果；
- 删除的重复或死代码；
- 有意保留的大文件及原因；
- 已运行的验证命令和结果；
- 是否修改契约、是否提交，以及剩余工作区状态。

## 完成标准

- 每个新文件或目录都有明确且稳定的职责；
- 共享规则只有一个权威实现，并由测试覆盖；
- 没有新增循环依赖、空壳抽象或含糊公共工具箱；
- 对外行为和生成契约保持不变，除非用户明确授权变更；
- 相关测试、质量门和差异复核通过；
- 无关修改得到保留，失败和未处理项被明确报告。
