# 任务问题审计

对本轮任务做一次只读、可追溯的执行审计，暴露曾影响执行判断的问题以及未处理事项，并在报告后停止等待用户决定。

## 边界

- 将审计范围限定为从最近一次启动当前目标的用户请求开始，到本 Skill 被触发时为止。纳入用户对同一目标的明确追加；目标已被明确替换时，不混入更早的无关任务。
- 默认只读取和核查证据，不修改文件、不安装工具、不启动或变更外部系统，也不实施修复方案。
- 可以查看当前上下文、工具错误、命令输出、日志、测试结果、工作区状态和差异，并重跑低风险、无外部副作用且与当前任务直接相关的验证。
- 不重跑可能写入数据或产生外部影响的命令。需要此类验证时，将其列为待执行项。
- 只报告可观察行为、简短决策依据和实际影响，不声称读取不可见的内部思维过程。

## 收录标准

收录所有实质性问题，包括已经解决的问题，只要它曾经导致以下任一结果：

- 改变原计划、实现路径或任务范围。
- 引入绕过、降级、fallback、重复状态、额外门控或不必要复杂度。
- 缩小验证范围、降低结果可信度或留下未经验证的假设。
- 因系统规则、用户规则、环境、权限或当前范围限制而未处理。
- 被主动判定为与当前任务无关并搁置，即使它没有阻塞当前交付。

省略没有实际影响且立即纠正的命令笔误。未经调查的偶然迹象只能标记为未知，不得扩写成已证实问题。

## 工作流

1. 确定审计范围。
   写明当前目标、起止边界、用户追加或替换过的范围，以及能够检查的证据来源。

2. 重建实际执行路径。
   对照用户目标与实际操作，查找失败、重试、路径变化、替代实现、跳过的检查、缩小的范围和主动搁置事项。不要因为最终命令成功而隐去此前问题。

3. 核验证据。
   优先引用错误输出、失败测试、日志、差异和明确执行记录。必要时执行边界允许的只读检查；证据不足时保留未知，不补全未经证实的故事。

4. 标记证据状态。
   - 已确认：有错误输出、失败测试、差异、日志或明确执行记录支持。
   - 推断风险：能从路径变化、复杂绕过或验证缺口合理推断，但尚未证实。
   - 未知：当前上下文不足以判断。

   同时标记处理状态为已解决、未解决、受限或范围外搁置，避免将“已经绕过”误写为“已经解决”。

5. 解释问题影响。
   对每个问题说明预期状态、实际发生的事情、当时采取的动作或绕过、未立即修复的原因，以及对正确性、完整性、验证可信度和后续维护的影响。

6. 形成根因修复方案。
   优先恢复最早被破坏的不变量，并将修复收敛到正确的责任所有者。明确应删除的绕过、fallback、重复状态和额外门控。只有根因修复受权限或范围限制时才提出临时方案，并标注限制原因、残余风险和删除条件。

7. 给出验证方式。
   为每项修复给出能够证明不变量成立的测试、命令或手动场景。区分本轮已执行的验证和仍待授权执行的验证。

8. 输出报告并停止。
   不自动恢复原任务，不实施任何修复。等待用户明确选择继续原任务、按方案修复或结束。

## 输出格式

按以下顺序输出；没有内容的部分明确写“无”：

1. **审计范围**：当前目标、时间边界和检查过的证据。
2. **总体结论**：问题数量、当前结果是否可信、是否存在未解决风险。
3. **当前任务问题**：逐项列出状态、证据、被破坏的不变量、当时采取的动作或绕过、未立即修复原因、影响、根因修复方案和验证方式。
4. **范围外发现**：列出主动搁置的问题、证据、范围判断、潜在影响，以及建议的责任所有者和继续处理条件。
5. **修复顺序**：按照阻塞正确性、数据或安全风险、验证可信度、结构复杂度的优先级排序。
6. **待用户决定**：只列需要新权限、扩大范围或产品取舍的事项。

如果没有发现实质性问题，明确报告“未发现实质性问题”，不得为了填充报告而虚构问题。
