# 沉淀项目知识

## 边界

- 默认只修改用户明确要求范围内的项目文档；未经明确要求，不提交、不推送，也不更新项目外部的个人记忆或外部系统。
- 只沉淀经过代码、测试、日志、契约或明确决策验证的稳定认知。一次性操作过程、可直接从代码读出的细节、临时路径、凭据和待验证假设不进入长期文档。
- 一个概念只选择一个权威所有者。已有文档职责合适时更新原文，不新建并行规则或兼容说明。

## 工作流

1. 收集证据。
   读取当前任务要求、相关差异、验证结果和现有文档，明确本轮真正新增或纠正了什么认知。

2. 判断是否值得长期保存。
   仅保留会影响未来实现、验证、排障或评审，并且预计会重复使用的规则。无法证明稳定性的内容明确排除或标记为待验证，不把猜测写成约定。

3. 选择权威位置。
   - 跨仓库个人偏好：只有用户明确要求时才更新个人全局指令。
   - 项目或目录行为、命令、验证和评审规则：更新作用域最近的 `AGENTS.md`。
   - 当前有效的架构、契约或维护方式：更新职责对应的项目文档。
   - 历史决策：仅在项目已有 ADR、CHANGELOG、迁移记录或事故复盘体系时按其职责记录。

4. 检查已有内容。
   在写入前搜索同一概念、近义规则和旧入口，选择修改或替换现有内容。新规则取代旧规则时删除过时说明，不保留两套可独立漂移的来源。

5. 写入可执行知识。
   使用简短规则说明适用范围、必须保持的不变量和验证方式。只有在能减少歧义时才加入命令、字段、错误行为或正反例；不记录冗长的任务经过。

6. 验证一致性。
   搜索是否仍有冲突或重复说明，运行仓库规定的文档、lint 或检查入口，并复核修改没有扩大到用户未授权范围。

7. 报告结果。
   列出更新的权威文档、沉淀的规则、主动排除的短期信息和实际验证结果。

## 知识质量标准

- 规则能够指导未来行动，而不只是描述已经发生的事情。
- 规则有明确作用域和所有者，不会覆盖更具体的项目约定。
- 规则与当前代码和验证结果一致，并说明必要的失败行为或检查方式。
- 文档只描述当前有效状态；需要保留历史时写入已有的历史型文档。
- 不包含凭据、个人隐私、机器专属临时值或能够从生成物重新得到的副本。
