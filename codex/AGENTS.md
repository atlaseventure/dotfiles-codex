## 工作方式

- 当前用户要求和当前路径下更具体的项目指令优先于个人全局默认；发生冲突时按更具体的要求执行并显式说明关键影响。
- 默认直接开始工作，在命令实际失败后再检查环境，不为尚未发生的问题预设回退路径。
- 仅当缺少的工具真正阻塞当前任务时进行安装；在受信任的 Unix 或 WSL 环境中可直接使用免密 `sudo`，其他平台使用对应的包管理方式。
- 文档、注释和面向用户的说明默认使用中文；用户要求、项目既有规范或外部格式另有规定时，以更具体的要求为准。

## 失败与根因

- 让失败通过错误、异常、日志和失败测试显式暴露，不模拟成功、不吞噬错误、不引入沉默默认值。
- 从最早被破坏的不变量追溯根因，优先删除冗余配置、重复状态、死分支和不必要门控，不在症状周围叠加旁路。
- 涉及共享验证、权限、路由、缓存、契约、迁移、跨模块状态或数据完整性时，按结构性问题处理，并让不变量只由一个明确的所有者维护。

## 执行边界

- 回答、解释、审查和诊断任务默认只读取与验证；实现任务完成检查、修改、验证和结果复核的完整闭环。
- 未经用户明确要求，不提交、推送、改写 Git 历史或扩大外部系统修改范围。
- 保留工作区中已有且与当前任务无关的修改；遇到重叠时先理解差异，不通过破坏性命令清理现场。

## 代码与文档

- 保持函数短小、控制流平坦、职责单一；只在复杂度确实存在时引入抽象、依赖注入或额外层次。
- 注释解释意图和权衡，不复述代码。维护型文档只描述当前有效状态；CHANGELOG、ADR、迁移记录和事故复盘等历史型文档按其职责保留历史。
- 一个概念只保留一个权威来源；其他位置通过引用或可验证的生成、安装关系使用它，不维护可独立漂移的副本。
